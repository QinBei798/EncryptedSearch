# EncryptedSearch 企业级可优化功能与架构升级指南

本项目在密码学算法、倒排索引及检索打分逻辑上已经具备了极高的完备性。为了让本系统从“单机算法原型”迈向真正的“企业级生产环境服务”，特规划以下五个维度的架构升级蓝图。

---

## 一、 并发读写与 I/O 底层重构 (Core Concurrency & I/O)

这是当前系统向高并发演进**最核心、性价比最高**的优化点。

1. **引入内存映射技术 (mmap)**
   * **问题**：目前使用 `std::ifstream::seekg` 和 `read` 在磁盘上定位并读取 Posting List。标准库的文件流在多线程并发执行 `seekg` 时是非线程安全的，会导致严重的数据竞争和崩溃。
   * **方案**：使用操作系统的内存映射机制（Linux 的 `mmap` 或 Windows 的 `CreateFileMapping`），将整个二进制索引文件直接映射到进程的虚拟内存空间。多线程可实现**无锁并发读取**，同时依靠 OS 的 Page Cache 达到接近内存级别的检索极速。

2. **增量索引与读写分离 (Delta Index & Merge)**
   * **问题**：生产环境不能接受为了添加新文件而重新全量扫盘构建索引。
   * **方案**：采用“双层索引”架构。新文件数据先写入内存中的 `DeltaIndex`，查询时同时查询磁盘 `MainIndex` 与内存 `DeltaIndex` 并合并结果。后台定时（或达到阈值时）加写锁将 `DeltaIndex` 刷盘，并与 `MainIndex` 进行归并重写（Merge）。

---

## 二、 服务化改造 (Service-oriented Architecture)

将命令行工具（CLI）改造为常驻内存的后端守护进程（Daemon）。

1. **RESTful API / 微服务封装**
   * 引入轻量级 C++ HTTP 框架（如 `cpp-httplib` 或 `oatpp`），或企业级 RPC 框架（如 `gRPC`、`brpc`）。
   * 暴漏供前端调用的 API 接口（接收 JSON 格式的查询请求，返回带有匹配路径和 TF-IDF 分数的 JSON 响应）。
2. **状态驻留机制**
   * 服务启动时一次性调用 `LoadIndex()` 完成词典的内存加载。随后主线程挂起并监听网络端口，实现真正的低延迟检索。

---

## 三、 极限数据与高并发压测 (Stress & Load Testing)

在上线前需摸清系统的物理瓶颈：

1. **海量数据构建压测（容量极限）**
   * 构建 10GB~100GB 级别的真实/模拟文档库。
   * 严密监控索引构建期间的内存峰值（验证 64KB 流式分词的防 OOM 效果），及评估构建耗时是否满足 RTO（恢复时间目标）。
2. **高并发检索压测（吞吐极限）**
   * 模拟 1000~10000 个高频并发请求，压测 QPS（每秒查询率）与 P99 尾部延迟。

---

## 四、 企业级安全性增强 (Enterprise Security Compliance)

针对等保三级及以上的高密级环境要求：

1. **密钥管理系统对接 (KMS)**
   * 摒弃用户手动输入密码派生密钥的方式，通过安全信道从企业硬件密码机（HSM）或统一密钥管理中心（如 HashiCorp Vault）动态拉取工作密钥。
2. **租户与权限隔离 (RBAC)**
   * 在底层 Posting 结构或 FileMap 中引入 `OwnerID` 或 `PermissionBitmask`。
   * 在 AST 查询树求值时（`EvaluateQuery`），强制加入权限过滤的位运算（AND 逻辑），确保用户只能检索到自身拥有权限的文档。

---

## 五、 容灾与可观测性 (Reliability & Observability)

1. **企业级异步日志**
   * 将标准输出（`std::cout/cerr`）全面替换为工业级异步日志库（如 `spdlog`）。
   * 支持日志轮转（Log Rotation）、多级别控制（DEBUG/INFO/WARN/ERROR）与多目标输出（控制台+文件）。
2. **原子性落盘 (Crash Consistency)**
   * 索引刷盘时，先写入临时文件 `index.bin.tmp`。写入彻底完成且 `flush` 后，利用操作系统提供的 `rename` 原语将其原子性地覆盖为正式的 `index.bin`，防止写一半断电导致整个系统瘫痪。
3. **监控与告警接入 (Metrics)**
   * 暴露 `/metrics` 采集接口。
   * 将引擎当前的运行状态（如内存占用、词典条目数、并发连接数、平均检索耗时）输出，对接 Prometheus 并在 Grafana 大屏上实现可视化。

---
> **实施建议（ROI 最高的第一步）**：优先使用 `cpp-httplib` 封装 HTTP 接口，并使用 `mmap` 解决并发读取问题，这两步改动极小，却能立刻让系统具备微服务的核心特征。
